## 一、数据库结构设计

### 1.1 ER 模型

系统涉及 4 个核心实体关系链：

```
Department(系) ──1:N── Major(专业) ──1:N── Class(班级) ──1:N── Student(学生)
Department(系) ──1:N── Teacher(教师)
Teacher(教师) ──1:N── Topic(题目)
Student(学生) ──1:1── Selection(选题) ──N:1── Topic(题目)
Selection(选题) ──1:N── Submission(提交) ──1:1── Review(审阅)
```

辅助实体：`Deadline`（流程期限配置）、`AuditLog`（审计日志，按时间分区）、`SysUser`（管理员）。

### 1.2 表结构清单（12 张表）

| # | 表名 | 主键 | 核心字段 | 说明 |
|---|------|------|----------|------|
| 1 | **department** | `dept_id` INT AUTO_INCREMENT | dept_name UNIQUE | 院系信息 |
| 2 | **major** | `major_id` INT AUTO_INCREMENT | major_name, dept_id (FK) | 专业信息，唯一约束 (name, dept) |
| 3 | **class** | `class_id` INT AUTO_INCREMENT | class_name, major_id (FK), grade | 班级信息，唯一约束 (name, major) |
| 4 | **student** | `stu_id` VARCHAR(20) | stu_name, password(BCrypt), class_id(FK), phone, email | 学生，含 CHECK 约束 |
| 5 | **teacher** | `teacher_id` VARCHAR(20) | teacher_name, password(BCrypt), dept_id(FK), title, research_direction, max_students | 教师，含 CHECK 约束 |
| 6 | **topic** | `topic_id` INT AUTO_INCREMENT | title, description(TEXT), teacher_id(FK), direction, status(0/1), max_select | 毕业设计题目 |
| 7 | **selection** | `selection_id` INT AUTO_INCREMENT | stu_id (UNIQUE FK), topic_id (FK), select_time, status, final_score | 选题记录，学生:题目=1:1 |
| 8 | **submission** | `submission_id` INT AUTO_INCREMENT | selection_id (FK), stage, file_path, status, version | 阶段文档，支持多版本 |
| 9 | **review** | `review_id` INT AUTO_INCREMENT | submission_id (UNIQUE FK), teacher_id(FK), score, comment | 审阅评分，每份提交仅一次 |
| 10 | **deadline** | `deadline_id` INT AUTO_INCREMENT | stage, start_date, end_date, semester | 流程期限，唯一约束 (stage, semester) |
| 11 | **audit_log** | `log_id` + `operation_time` | table_name, operation, record_id, old_value(JSON), new_value(JSON) | 审计日志，**按年分区** |
| 12 | **sys_user** | `user_id` INT AUTO_INCREMENT | username UNIQUE, password(BCrypt), role | 管理员账户 |

### 1.3 关系与约束

- **外键约束**：所有关联表均设置 FOREIGN KEY，级联策略根据业务语义选择：
  - `CASCADE`：院系→专业→班级（删除院系时级联清理）
  - `RESTRICT`：学生/教师/选题相关（有数据时禁止删除）
- **CHECK 约束**：手机号长度、邮箱格式、成绩范围(0-100)、题目状态(0/1)、流程阶段枚举值
- **UNIQUE 约束**：选题 `stu_id`（一生一题）、审阅 `submission_id`（一提交一审）、期限 `(stage, semester)`
- **默认值**：题目状态=0(可选)、选题状态='进行中'、提审状态='待审阅'、教师最大指导学生=5

### 1.4 索引设计

| 类型 | 索引名 | 表.列 | 用途 |
|------|--------|------|------|
| 主键索引 | PRIMARY | 各表主键 | 自动创建，聚簇索引 |
| 唯一索引 | uk_student_email | student.email | 邮箱唯一 |
| 唯一索引 | uk_teacher_email | teacher.email | 邮箱唯一 |
| 普通索引 | idx_student_class | student.class_id | 按班级查学生 |
| 普通索引 | idx_topic_teacher_status | topic(teacher_id, status) | 教师查可选题目 |
| 普通索引 | idx_submission_sel_stage_status | submission(selection_id, stage, status) | 多条件查提交 |
| **全文索引** | ft_topic_title | topic.title | 题目模糊搜索（ngram 解析器） |
| **全文索引** | ft_topic_desc | topic.description | 描述全文搜索 |

## 二、8 项高阶数据库技术实现

### 技术 1：存储过程（Stored Procedure）

**应用场景**：成绩加权计算、院系完成率统计

#### 存储过程 1 — `calc_final_score`（加权计算最终成绩）

```sql
PROCEDURE calc_final_score(
    IN  p_stu_id    VARCHAR(20),   -- 输入：学号
    OUT p_final_score DECIMAL(5,2), -- 输出：综合成绩
    OUT p_status    VARCHAR(50),    -- 输出：完成状态
    OUT p_detail    VARCHAR(500)    -- 输出：成绩明细
)
```

**核心逻辑**：查询学生四个阶段（开题/中期/初稿/终稿）最新版本的审阅成绩，按权重加权求和：

```
综合成绩 = 开题×15% + 中期×15% + 初稿×30% + 终稿×40%
```

同时根据各阶段状态判断毕设进度（已选题→开题阶段→中期阶段→论文撰写→终稿审阅→已完成），并将结果写回 `selection.final_score`。

#### 存储过程 2 — `stat_dept_completion`（院系完成情况统计）

```sql
PROCEDURE stat_dept_completion(
    IN  p_dept_id INT,          -- NULL=全校统计
    OUT p_total_stu INT,         -- 学生总数
    OUT p_completed INT,         -- 已完成数
    OUT p_no_select INT,         -- 未选题数
    OUT p_report VARCHAR(1000)   -- 统计报告
)
```

**核心逻辑**：跨 student → class → major → department 四级 JOIN，统计某系毕业设计完成率，传入 NULL 则全校统计。

### 技术 2：触发器（Trigger）

**应用场景**：自动化数据联动，共 3 个触发器

#### 触发器 1 — `trg_selection_after_insert`

```
时机：AFTER INSERT ON selection
作用：学生选题后自动更新题目状态
```

实现：统计题目已选人数，若达到 `max_select` 上限，自动将 `topic.status` 设为 1（已选），防止超选。

#### 触发器 2 — `trg_submission_audit_log`

```
时机：AFTER INSERT ON submission
作用：文档提交时自动写入审计日志
```

实现：将提交信息（selection_id、stage、file_path、status、version）封装为 JSON 对象，写入 `audit_log` 表，`operated_by` 记录操作学生学号。

#### 触发器 3 — `trg_review_after_insert`

```
时机：AFTER INSERT ON review
作用：教师评分后自动更新状态 + 重算成绩
```

实现：
1. 根据评分判断（≥60 通过，<60 需修改），更新 `submission.status`
2. 内联重算四个阶段的加权综合成绩，更新 `selection.final_score`
3. 写入审计日志（JSON 格式记录评分详情）

> **注意**：MySQL 触发器中不能直接 CALL 带 OUT 参数的过程，故在触发器内部使用子查询内联计算成绩。

### 技术 3：视图（View）

**应用场景**：简化复杂查询、隐藏敏感字段、提供统一数据接口

#### 视图 1 — `v_student_progress`（学生进度总览）

多表 JOIN（student → class → major → department → selection → topic → teacher），并用 12 个子查询展示每个学生四阶段的提交状态、时间和成绩。一行即可查看学生完整毕设进度。

#### 视图 2 — `v_topic_detail`（选题详情）

JOIN teacher + selection + student + class + major 六张表，提供「题目→教师→学生」完整信息链，暴露教师联系方式但**隐藏学生密码字段**。

#### 视图 3 — `v_score_ranking`（成绩排名基础数据）

提供成绩排名所需的全量多表 JOIN 结果，作为窗口函数查询的**数据基础**。

### 技术 4：索引优化

**应用场景**：高频查询加速、全文搜索支持

#### 全文索引（FULLTEXT INDEX）

为 `topic.title` 和 `topic.description` 创建全文索引，使用 **ngram 解析器**（支持中文分词）：

```sql
ALTER TABLE topic ADD FULLTEXT INDEX ft_topic_title (title) WITH PARSER ngram;
```

查询对比：

| 查询方式 | SQL | 扫描类型 | 预估耗时 |
|----------|-----|----------|----------|
| LIKE 模糊 | `WHERE title LIKE '%毕业设计%'` | ALL（全表扫描） | ~50ms |
| 全文索引 | `WHERE MATCH(title) AGAINST('毕业设计')` | fulltext | ~3ms |

#### 复合索引

针对多条件高频查询创建复合索引，如：
- `idx_submission_sel_stage_status (selection_id, stage, status)` — 覆盖「某选题 + 某阶段 + 某状态」三维查询
- `idx_topic_teacher_status (teacher_id, status)` — 教师查看某状态题目

### 技术 5：事务与并发控制

**应用场景**：学生选题场景，防止同一题目被多人同时选择

#### 悲观锁实现

在 `SelectionServiceImpl.selectTopic()` 方法中使用 `@Transactional` 注解：

```java
@Transactional
public Map<String, Object> selectTopic(String stuId, Integer topicId) {
    // 1. 检查 deadline 窗口
    // 2. 检查学生是否已选题（SELECT COUNT）
    // 3. 检查题目状态是否为可选（status=0）
    // 4. 检查教师容量
    // 5. UPDATE topic SET status=1  ← 关键：先更新题目状态
    // 6. INSERT selection              ← 再插入选题记录
    // 如果并发，步骤5的UPDATE会获取行锁，
    // 第二个事务在步骤3读到status=1，直接返回"已选"
}
```

**并发场景**：两个学生同时选择同一题目时，先执行 UPDATE 的事务获取行级锁，第二个事务在锁释放后重新读取，发现 status 已变为 1，返回错误提示"该题目已被选择"。

### 技术 6：窗口函数（Window Functions）

**应用场景**：成绩排名、教师统计、分段分析，共 7 个查询

| 查询 | 窗口函数 | 应用 |
|------|----------|------|
| 系内/专业内成绩排名 | `DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY score DESC)` | 按系和专业分组排名 |
| 教师指导学生数排名 | `RANK() OVER (ORDER BY COUNT(*) DESC)` | 教师工作量排名 |
| 各专业最高/最低分 | `ROW_NUMBER() OVER (PARTITION BY major_id ORDER BY score DESC/ASC)` | 极值学生提取 |
| 成绩累积分布 | `CUME_DIST() OVER (ORDER BY score DESC)` | 每位学生超过xx%同学 |
| 提交时间间隔 | `LAG(submit_time, 1) OVER (PARTITION BY selection_id, stage ORDER BY version)` | 相邻版本提交间隔 |
| 各系学生占比 | `SUM(COUNT(*)) OVER ()` | 全校学生分布 |
| 阶段通过率 | GROUP BY + 聚合 CASE | 四阶段通过率统计 |

示例 SQL（系内排名）：
```sql
SELECT d.dept_name, s.stu_name, sel.final_score,
       DENSE_RANK() OVER (PARTITION BY d.dept_id ORDER BY sel.final_score DESC) AS 系内排名
FROM student s
JOIN selection sel ON s.stu_id = sel.stu_id
JOIN ...;
```

### 技术 7：权限管理（GRANT/REVOKE）

**应用场景**：数据库层面最小权限原则，创建 3 个 MySQL 角色

| 角色 | 用户名 | 权限范围 |
|------|--------|----------|
| 学生 | `gms_student@localhost` | SELECT 题目/教师/期限/视图；SELECT+INSERT 选题/提交；SELECT 评阅；UPDATE 个人信息（部分列） |
| 教师 | `gms_teacher@localhost` | 学生基础上 + INSERT/UPDATE/DELETE 题目；INSERT+UPDATE 审阅；SELECT 审计日志 |
| 管理员 | `gms_admin@localhost` | ALL PRIVILEGES（不含审计日志的 INSERT/UPDATE，由触发器自动维护） |

**权限粒度示例**（列级权限）：
```sql
-- 学生只能修改自己的手机、邮箱、密码，不能改姓名和班级
GRANT SELECT, UPDATE (phone, email, password)
    ON graduation_management.student TO 'gms_student'@'localhost';

-- 教师只能查看学生基础信息，不暴露密码
GRANT SELECT ON graduation_management.student TO 'gms_teacher'@'localhost';
```

> **实际部署**：应用层连接池统一使用 root 账号，RBAC 在应用层实现。以上 MySQL 用户主要用于演示 GRANT/REVOKE 技术和运维场景权限隔离。

### 技术 8：审计日志（Audit Log）

**应用场景**：追踪所有关键操作，满足数据安全与合规要求

#### 表结构设计

`audit_log` 表使用 **JSON 列**存储操作前后的数据快照：

| 字段 | 类型 | 说明 |
|------|------|------|
| table_name | VARCHAR(50) | 操作的表名 |
| operation | VARCHAR(10) | INSERT / UPDATE / DELETE |
| record_id | VARCHAR(50) | 被操作记录的主键 |
| old_value | JSON | 操作前数据（JSON） |
| new_value | JSON | 操作后数据（JSON） |
| operated_by | VARCHAR(30) | 操作者ID |
| operation_time | DATETIME | 操作时间 |
| ip_address | VARCHAR(50) | 操作IP |

#### 分区策略

```sql
PARTITION BY RANGE (TO_DAYS(operation_time)) (
    PARTITION p2024 VALUES LESS THAN (TO_DAYS('2025-01-01')),
    PARTITION p2025 VALUES LESS THAN (TO_DAYS('2026-01-01')),
    PARTITION p2026 VALUES LESS THAN (TO_DAYS('2027-01-01')),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

按年度分区，**提升按时间段查询性能**（分区剪枝），**便于历史数据归档**（直接 DROP 旧分区）。

#### 写入机制

由**触发器自动写入**（参见技术 2），无需应用层手动调用，对业务代码透明。关键操作数据以 JSON 格式存储，灵活支持半结构化日志。
